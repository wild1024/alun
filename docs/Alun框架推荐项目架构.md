基于Alun 框架针对“AI辅助开发+快速迭代”的场景，推荐**简洁且兼顾功能完整性**的分层方案是：

> **按业务模块垂直划分，每个模块内只保留 handler / dto / model 三层**，所有**横切关注点**（中间件、任务、插件、工具类）**在模块外**统一管理。

这样既保持了业务高内聚，又避免在模块内部堆砌与业务无关的代码，让AI和开发者都能一眼看清模块的核心逻辑。

## 📌 各横切关注点的归属说明

### 1. 中间件 → `middleware/`
- **职责**：HTTP 请求前/后处理的通用逻辑（框架无法满足的自定义中间件等）。
- **放置原因**：与具体业务模块无关，所有模块共用。
- **调用方式**：在 `main.rs` 或路由层全局注册，不是业务模块自身关心的事。

### 2. 后台任务 → `task/`
- **职责**：异步、定时或队列驱动的任务（如导出报表、发送欢迎邮件、同步库存）。
- **放置原因**：这些任务通常由**事件或调度器触发**，可以引用框架提供的task插件，执行时可能调用多个业务模块的服务。放在模块外避免循环依赖。
- **使用方式**：`task/report_task.rs` 内部调用 `modules/order/hanlder`。

### 3. 插件 → `plugins/`
- **职责**：第三方集成的可插拔能力（支付网关、短信通道、文件存储）。
- **放置原因**：这些通常需要**统一接口 + 配置切换**（例如从阿里云短信换成腾讯云）。独立出来方便热插拔和测试。
- **使用方式**：业务模块通过 `plugins::payment::pay(order)` 调用，不直接依赖具体实现。

### 4. 工具类 → `utils/`
- **职责**：纯函数式的辅助工具（日期格式化、随机字符串）。
- **放置原因**：无状态、无副作用，可被任何模块随意调用，放在顶层最自然。

### 5. 共享类型 → `shared/`
- **职责**：自定义配置结构体 `AppConfig`、常量定义等。
- **放置原因**：避免在各个模块重复定义，同时避免循环引用。

## 🧩 业务模块内部（handler / dto / model）的清晰分工

| 文件           | 职责                                                              | 是否依赖外层                                                   |
| ------------ | --------------------------------------------------------------- | -------------------------------------------------------- |
| `handler.rs` | 路由函数：解析请求 →业务处理-> 调用 model 的数据库处理方法 → 构造响应                      | ✅ 可依赖 `model`、`dto`、`shared`、`utils`、`middleware`（仅类型依赖） |
| `dto.rs`     | 请求结构（`CreateUserReq`）和响应结构（`UserRes`），可包含 `serde` 和 `validator` | ❌ 一般不依赖其他模块（可依赖 `shared` 的基础类型）                          |
| `model.rs`   | **核心**：实体定义、数据库操作（Repository）、返回结构体封装                           | ✅ 可依赖 `shared`、`utils`、`plugins`、其他模块的 `model`（经过接口）     |

> 📎 注：在这个极简方案中，**没有独立的 `service` 层**。业务逻辑直接放在 `handler.rs` 的方法中，然后调用 `Model` 的impl 操作方法 实现。这足够简洁，且适合 AI 生成代码 —— 因为 AI 理解“模型是数据+行为”比理解额外的抽象层更容易。

---

## ⚙️ 如何避免模块间耦合？

在 `model.rs` 中只做数据库操作，牵扯到通用业务操作抽离成单独业务函数互相调用。
数据库操作可以直接用 `modules::inventory::model` 中的函数 —— 对于中小型 ERP 来说这种直接耦合是可以接受的，因为模块本身就是按业务拆的。

---

## ✅ 总结：这个方案的优点

1. **极简**：每个业务模块只有 3 个文件（handler / dto / model），没有多余的 service/dao/mapper 层。
2. **AI 友好**：文件命名和职责高度一致，AI 可以根据路径和文件名准确推测应该写什么代码。
3. **横切关注点不污染业务**：中间件、任务、插件、工具类都在顶层，业务模块只需 import 它们。
4. **容易扩展**：当模块复杂度上升，可以轻松将 `handler.rs` 拆成 `service.rs` + `model.rs`，而不改变目录骨架。

这个结构已经被用在多个中小型 Rust Web 项目中，非常适合 **AI 辅助开发 + 人力有限 + 业务快速变化** 的 常见业务 系统初期阶段。
